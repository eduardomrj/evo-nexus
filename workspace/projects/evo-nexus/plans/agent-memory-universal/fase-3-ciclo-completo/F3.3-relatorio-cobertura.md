# F3.3 — Relatório de cobertura de memória

**Tipo:** [EVOLUIR]
**Agentes envolvidos:** @bolt-executor, @clawdia-assistant (EOD)
**Complexidade:** LOW

## Descrição

Adicionar ao daily log do EOD uma seção que mede a saúde do sistema de memória: quantos agentes ativos no dia salvaram proativamente (A), quantos foram cobertos pela rede de segurança (B), e quantos precisaram da extração de recuperação (C). É o instrumento que diz se a Camada A está funcionando e se vale promover o modelo de haiku para sonnet (decisão de F1.4).

## Passos concretos

1. No fluxo do EOD, após F3.2, cruzar:
   - Agentes ativos no dia (F3.1).
   - Quais tinham memória salva ANTES da extração de recuperação (A ou B).
   - Quais só ganharam memória via F3.2 (C).
2. Emitir tabela no daily log:
   ```
   | Agente | Ativo hoje | Salvou (A) | Rede B | Recuperação C | Entradas novas |
   ```
3. Sinalizar tendência: se muitos caem em B/C, a Camada A (proativa) não está pegando → revisar prompts. Se a qualidade das entradas de C for baixa → considerar promover modelo (F1.4).

## Acceptance criteria

- Daily log do EOD passa a conter a tabela de cobertura.
- A tabela distingue corretamente A vs B vs C.
- Métrica de custo da extração do dia aparece no log.

## Dependências

F3.1 e F3.2.

## Riscos

- BAIXO: ruído no log se a tabela ficar grande. Mitigar mostrando só agentes ativos no dia.
