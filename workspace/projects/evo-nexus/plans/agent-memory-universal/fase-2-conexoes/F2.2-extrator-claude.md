# F2.2 — Extrator de memória (invoca Claude)

**Tipo:** [CONSTRUIR NOVO]
**Agentes envolvidos:** @bolt-executor (implementa), @apex-architect (revisa contrato), @oath-verifier (verifica fail-open)
**Complexidade:** MEDIUM

## Descrição

Script Python que recebe (slug, transcript) e produz entradas de memória tipadas, gravando em `.claude/agent-memory/{slug}/` no formato canônico (frontmatter + linha no `MEMORY.md`). É o **núcleo reutilizado pelas Camadas B e C** — F2.1 e F3.2 ambos chamam este script. Segue a convenção de projetos customizados do CLAUDE.md: código em local versionável, dados em `evo-projects-data/`.

## Passos concretos

1. Criar `ADWs/routines/evo-projects/agent-memory-extract.py` (ou em `plugins/agent-memory/`), expondo uma função `extract(slug, transcript_text) -> list[MemoryEntry]`.
2. Aplicar o **gate de privacidade** decidido em F1.4 ANTES de chamar o modelo:
   - Se o transcript bate nos marcadores de segredo → pular sessão, logar `skipped_privacy`, exit 0.
3. Montar o prompt de extração para o modelo de F1.4 (haiku) com:
   - Os 4 tipos canônicos e quando salvar cada (user/feedback/project/reference).
   - A lista do que NÃO salvar (código, git history, fix recipes, conteúdo de CLAUDE.md, efêmero, **segredos**).
   - Instrução de redação de credenciais (gate (b) de F1.4).
   - Saída estruturada: JSON `[{name, description, type, body}]`.
4. **Deduplicar** contra o `MEMORY.md` existente do slug: ler o índice, comparar por `name`/descrição, descartar duplicatas (regra "não duplicar" do protocolo).
5. Para cada entrada nova: gravar arquivo `{tipo}_{slug-curto}.md` com frontmatter e adicionar **uma linha** ao `MEMORY.md` (nunca conteúdo no índice).
6. Invocar via `EvoClient`/CLI do Claude já usado nas rotinas; logar custo/tokens.

## Acceptance criteria

- Dado um transcript de exemplo, gera entradas tipadas válidas (frontmatter correto) em `.claude/agent-memory/{slug}/`.
- Sessão com segredo → nenhuma entrada gravada, log `skipped_privacy`.
- Entrada já existente no `MEMORY.md` → não duplicada.
- `MEMORY.md` recebe só linha de índice, nunca corpo de memória.
- Falha do modelo/timeout → exit 0, nenhuma escrita parcial corrompendo arquivos.

## Dependências

**F1.4** (modelo + gate). Reutilizado por F2.1 (Camada B) e F3.2 (Camada C).

## Riscos

- MÉDIO: extração de baixa qualidade poluir memória. Mitigar com prompt rígido + dedup + Memory Lint semanal.
- ALTO se F1.4 ignorado: vazar segredo. Gate é pré-condição obrigatória.
- BAIXO: escrita concorrente no mesmo `MEMORY.md` (B e C ao mesmo tempo). Mitigar com lock de arquivo simples ou append atômico.
