# Work Plan — Discord OpenClaude Bridge v1.5

## Context

O runtime ativo do Discord OpenClaude Bridge é a v1 em `/home/evonexus/evo-projects/discord-openclaude-bridge`; a v2 permanece congelada e o MOD-07 já foi mergeado no `main` (`2042577 Merge pull request #10 from eduardomrj/refactor/mod-07`). O bug recorrente `Separator is found, but chunk is longer than limit` aparece em execuções com subagentes antes do chunker final, como na execução `487f6588-8a38-4622-820c-5090bddd0b3a` (`project_slug=cpsmq`, `current_agent=bolt-executor`, `current_tool=Read`, `tool_counts Agent=2/Read=13`).

A solução aprovada por Apex é a solução C em formato v1.5: introduzir `bridge_reply` controlado, preservar Agent/Skill/MCP, registrar `reply_called`/`reply_chunks_sent`, impedir duplicação de `result.text` e manter fallback legado controlado.

## Objectives

- Permitir que execuções longas enviem resposta final via `bridge_reply` controlado sem passar a carga inteira pelo `result.text` do OpenClaude.
- Preservar o uso atual de Agent, Skill e MCP; nenhuma proposta deve reduzir permissões, remover ferramentas ou trocar a arquitetura por bridge v2.
- Garantir compatibilidade com comandos atuais (`/status`, `/last`, `/cancel`, `/project`, `/model`, `/start`, `/reset-session`) e com o fluxo de execução assíncrono atual.
- Adicionar telemetria testável: eventos `reply_called`, `reply_chunks_sent`, e fallback explícito quando `bridge_reply` não for usado.

## Guardrails

### Must Have

- Menor diff seguro por PR, mantendo a v1 como runtime ativo.
- `Agent`, `Skill` e MCP devem continuar permitidos no modo de execução atual; a v1.5 adiciona um canal de resposta, não substitui a linha de execução.
- O envio live/smoke no Discord fica fora da implementação local e só ocorre em etapa separada com aprovação.
- Não reiniciar serviço durante a implementação local.
- Fallback legado deve continuar entregando `result.text` quando `bridge_reply` não foi chamado.

### Must NOT Have

- Não remover `Agent`, `Skill`, MCP, `--tools default`, permissões atuais ou suporte a subagentes.
- Não migrar para v2 nem reabrir decisões da v2 congelada.
- Não alterar sem necessidade comandos de usuário, registry de projetos ou política de acesso.
- Não fazer smoke live, restart de systemd, ou uso real do bot como parte dos PRs locais.

## Recommendation — Ordem de PRs

Recomendo PRs sequenciais pequenos, não um PR grande.

Motivo: o risco real está no acoplamento entre OpenClaude subprocess, MCP/tools e entrega Discord. Separar em 4 PRs permite validar isoladamente: primeiro o contrato de envio, depois a exposição ao runner, depois a lógica anti-duplicação/fallback, por fim documentação/gate. Um PR grande dificultaria identificar se uma regressão veio do MCP, do chunking, do parser ou do handler de sucesso.

Ordem recomendada:

1. PR 1 — contrato local de `bridge_reply` e logs.
2. PR 2 — exposição ao runner preservando Agent/Skill/MCP.
3. PR 3 — anti-duplicação e fallback legado controlado.
4. PR 4 — docs, gate local e preparação de smoke live aprovado.
