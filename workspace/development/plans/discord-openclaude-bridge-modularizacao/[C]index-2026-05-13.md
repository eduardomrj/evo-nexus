# Modularização do Discord OpenClaude Bridge

**Tipo:** índice. Cada item tem arquivo próprio na pasta da respectiva fase. Este arquivo não duplica o detalhamento — aponta pros arquivos-filhos.

**Estratégia-base:** baseline validado em 2026-05-13 + revisão Compass/Raven para modularização incremental.

---

## Contexto

O `discord-openclaude-bridge` está funcional e validado em smoke/testes até o momento. O risco atual é arquitetural: o arquivo `src/discord_openclaude_bridge.py` tem cerca de 3.400 linhas e o `BridgeHandler` concentra allowlist, comandos, onboarding, sessão, concorrência, runner, logging e UX.

Este plano organiza a modularização sem mudança funcional, sem restart de serviço e sem implementação de novas features nesta onda.

## Objetivos

- Reduzir o `BridgeHandler` para uma facade fina de coordenação.
- Preservar comportamento externo dos comandos e do runtime validado.
- Separar contratos, routing, persistência, runner, progresso, adapter Discord e handlers de comando.
- Preparar a base para futuras features (`/skills`, `add_dirs`, `/project new`) sem implementá-las agora.

## Guardrails

**Must Have**
- Preservar `/start`, `/reset-session`, `/status`, `/context`, `/last`, `/project`, allowlist multi-canal, thread onboarding e propagação de `add_dirs`.
- Congelar comportamento externo antes de mover código.
- Preservar compatibilidade SQLite/runtime e redaction/logging.
- Rodar verificação automatizada após cada extração relevante.
- Pedir aprovação antes de qualquer smoke manual no Discord ou ação operacional.

**Must NOT Have**
- Não reiniciar serviços como parte do plano.
- Não limpar worktrees existentes sem decisão explícita.
- Não implementar `/skills`, `/project new` completo ou gestão avançada de `add_dirs` nesta onda.
- Não mudar UX pública, slash tree, schema persistido ou `projects.yaml` sem critério de aceite explícito.

- **Gate 0:** repo limpo, baseline validado, MOD-00 pronto, MOD-01 aprovado, worktree decidido, estratégia de branch/PR definida, nenhum runtime/secret incluído.
- **Gate 1:** `py_compile`, import smoke, testes focados, suíte completa, extrações de baixo risco concluídas, sem restart, sem mudança funcional.

- **Gate 3:** `BridgeHandler` facade fina, docs atualizadas, seams documentados sem feature behavior change, verificação final por @oath e smoke Discord final com aprovação.
