## Decisão arquitetural

O projeto deve continuar sendo um único serviço:

- 1 serviço systemd
- 1 processo Python
- 1 deploy
- 1 runtime DB/logs
- sem microserviços

A mudança desejada é sair de um monolito de arquivo:

```text
src/discord_openclaude_bridge.py
```

para um pacote Python modular interno, mantendo o arquivo antigo como shim/fachada compatível durante a transição.

## Estrutura alvo

```text
src/discord_openclaude_bridge.py        # shim/fachada temporária

src/discord_openclaude_bridge/
  __init__.py
  __main__.py
  cli.py
  config.py
  models.py        # ou types.py
  store.py
  runner.py
  stream_parser.py # ou stream.py
  handler.py
  discord_io.py
  client.py

  security/
    __init__.py
    redaction.py
  formatting/
    __init__.py
    chunking.py
  telemetry/
    __init__.py
    logging.py
```

## Guardrails

Não fazer nesta modularização inicial:

- microserviços;
- novo daemon/processo;
- alteração de systemd;
- alteração de env/secrets/Vaultwarden;
- alteração de DB/runtime path;
- mudança de comandos Discord;
- mudança de UX;
- project routing;
- redesign de schema SQLite;
- decompor internamente o `BridgeHandler` antes de extrair módulos estáveis.

Manter compatibilidade com:

- `python3 src/discord_openclaude_bridge.py --check-config`;
- import legado usado pelos testes;
- launcher atual;
- API pública atualmente usada nos testes.

## Diagnóstico arquitetural

O arquivo atual é um monolito funcional, mas com fronteiras conceituais claras:

1. Configuração e defaults operacionais.
2. Tipos/enums/dataclasses de domínio.
3. Persistência SQLite.
4. Sanitização/redaction.
5. Runner OpenClaude.
6. Parser de stream JSON.
7. Adapters/helpers Discord.
8. `BridgeHandler`.
9. Client Discord e slash commands.
10. CLI/main.

O maior risco é o `BridgeHandler`, porque concentra:

- Discord UX;
- sessões por canal/tópico;
- concorrência;
- `/cancel`;
- `/reset-session`;
- runner;
- persistência;
- milestones/reactions;
- status/cost/context.

Portanto: **não começar pelo handler**.

## Riscos principais

| Risco | Mitigação |
|---|---|
| Import circular | `models/types.py` não importa módulos de alto nível; dependências sempre para baixo. |
| Quebrar launcher | Manter `src/discord_openclaude_bridge.py` como shim até o final. |
| Quebrar reset/session race | Não alterar ordem de persistência sem teste específico. |
| Quebrar cancelamento | Proteger `pid/pgid` e process group com testes. |
| Expor segredo em telemetry | Extrair redaction cedo e manter testes de regressão. |
| Spam Discord | Manter testes de low-noise ACK/reactions. |
| Tocar DB produtivo | Testes sempre em `tmp_path`; nunca usar runtime data nos testes. |
